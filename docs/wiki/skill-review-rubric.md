# Skill Review Rubric

Phoebe does not want a score-first gallery. The public recommendation method is evidence-first and scenario-based.

## Required Public Notes

Each published Skill should capture:

| Field | Question |
| --- | --- |
| Core concept coverage | What are the Skill's main functions, methods, or concepts, and did the test use about 80% of them? |
| Ground-up artifact | What did Phoebe build from scratch using the Skill, not merely tweak or plan? |
| Rating | What is the high-level rating out of 10 after the hands-on build? |
| 3 good points | What made the Skill genuinely useful in practice? |
| 3 can-be-better points | What friction, limits, or improvement areas did Phoebe notice? |
| Use cases | How could data people, AI builders, C-suite readers, students, or automation learners use it day to day? |
| Demo evidence | What product, demo, flowchart, mindmap, screenshot, or critique was produced while testing it? |
| Source credibility | What GitHub repo or reference path is the Skill from, and does it have at least 10k stars or verified Skill of the Day, Repo of the Day, top repo, or equivalent ranking proof? |
| Recommendation decision | Is it published, still exploring, on the wishlist, or rejected quietly? |

## 80% Ground-Up Skill Trial

Skill exploration is a field test, not a light impression. For each Skill Phoebe explores:

1. Read the Skill source and identify its key functions, methods, or concepts.
2. Choose an original artifact to build from the ground up with that Skill.
3. Use roughly 80% of the Skill's key concepts during the build. If a concept is skipped, note why.
4. Keep proof of the build: page, demo, workflow, diagram, mindmap, report, screenshot, or repo change.
5. Give a high-level rating out of 10 after the build.
6. Capture exactly 3 things the Skill does really well.
7. Capture exactly 3 things that can be further improved.
8. Verify the source credibility gate: at least 10k GitHub stars, or verified Skill of the Day, Repo of the Day, top repo, or equivalent ranking proof.
9. Decide whether it becomes a public pick, stays exploring, or gets rejected quietly.

Do not publish a Skill review based only on reading, installing, planning, or using it for a small amendment. The Skill must help produce something new from zero.

## Demo Artifact Direction

The demo page is part of the evaluation, not a decorative subpage. It should showcase what the Skill can actually do and use as many relevant Skill features as practical.

Every explored Skill subpage should list the Skill's key features in a visual, interactive format. Treat the features as controls, cards, branches, maps, diagrams, toggles, or live states so followers can see what the Skill changes in practice. Avoid hiding the feature list inside long prose.

Do not make every demo look the same. A Skill demo may become a product prototype, decision flow, mindmap, diagram, interactive lab, visual critique, dashboard, report, or any other static artifact that best expresses the Skill's function. Keep the gallery homepage consistent for browsing, but let `demos/` pages vary widely when variety helps followers understand the Skill.

## Status Labels

| Data value | Public label | Meaning |
| --- | --- | --- |
| `wishlist` | Wishlist | Worth trying, not tested yet. |
| `exploring` | Exploring | Currently being used in a real build or evaluation. |
| `published` | Published | Useful enough to recommend publicly. |
| `rejected` | Rejected | Tried, but not worth recommending right now. Keep most rejected Skills out of the public gallery. |

## Promotion Rules

Promote a Skill to `published` only when it has:

- A ground-up artifact built with the Skill.
- Evidence that the test used about 80% of the Skill's key concepts, or a clear note explaining the gaps.
- A high-level rating out of 10.
- A real hands-on test.
- A proof artifact or demo.
- Clear value for reducing repeated prompting or improving AI-assisted work.
- 3 practical good points.
- 3 honest improvement points.
- At least three day-to-day use cases for the target audience.
- A source URL, source repo/reference path, and dated GitHub star snapshot when the source is on GitHub.
- At least 10k GitHub stars, or verified Skill of the Day, Repo of the Day, top repo, or equivalent ranking proof.
- Acceptable safety and maintenance profile.
